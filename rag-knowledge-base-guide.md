# 企业知识库 RAG 方案详解

> 基于 2024-2026 年最佳实践整理

---

## 一、RAG 是什么？

### 传统 LLM 的问题
- 会"幻觉"（编造信息）
- 不知道公司的私有数据
- 知识截止于训练时间

### RAG 的解决方案
不让 LLM 靠记忆回答，而是先从文档里检索相关内容，再基于检索结果生成答案。

**核心流程：**
```
用户提问 → 检索知识库 → 找到相关文档片段 → 连同问题一起喂给 LLM → 生成答案
```

### RAG 的好处
- 答案有据可查（可追溯来源）
- 减少幻觉
- 知识库更新后立即生效，不用重新训练模型

---

## 二、整体架构

### 知识库搭建阶段
```
原始文档 → 文档解析 → 切块(Chunking) → 生成嵌入(Embedding) → 存入向量数据库
(PDF/Word)   (OCR/解析)                              (Vector DB)
```

### 问答使用阶段
```
用户提问 → 问题嵌入(Embedding) → 向量检索(Search) → 重排序(Rerank) → 拼接上下文(Context) → LLM生成(Answer)
```

---

## 三、核心组件详解

### 1. 文档解析

**挑战：**
- PDF 有扫描件、表格、多栏排版
- Word 有复杂格式
- 网页有 HTML 结构

**处理方式：**
- 简单文档：直接提取文本
- 复杂文档：需要 OCR + 布局理解
- 表格：保留结构，转成 Markdown 或 JSON

**工具差异：**

| 工具 | 文档解析能力 |
|------|-------------|
| Dify | 基础良好，支持 12+ 格式 |
| FastGPT | 中等，9 种格式 |
| **RAGFlow** | **最强**（DeepDoc 技术，擅长扫描件、表格、复杂布局） |

---

### 2. 切块（Chunking）

**为什么需要切块？**
- 文档太长，不能整篇塞给 LLM
- 需要精确匹配问题相关的片段

**切块策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 固定大小 | 按 token 数切（如 512 tokens） | 简单文档，快速原型 |
| **递归切块** | 按段落→句子→词语层级切 | **生产推荐起点** |
| 语义切块 | 按语义相似度聚合成块 | 追求精度，但计算成本高 |
| 结构感知 | 按标题、章节、表格边界切 | 企业文档（手册、政策） |
| 层级切块 | 小块检索，大块返回上下文 | 复杂文档，需要上下文 |

**最佳实践：**
- 块大小：**400-512 tokens**
- 重叠：**10-20%**（避免边界信息丢失）
- 保留结构：在块前加上文档标题/章节名

---

### 3. 嵌入（Embedding）

**什么是嵌入？**
把文本转成高维向量（如 768 维），语义相似的文本向量距离近。

**嵌入模型推荐：**

| 模型 | 特点 |
|------|------|
| **BGE-M3** | 开源，中英文效果好，**首选** |
| Cohere Embed | 商业方案，企业友好 |
| OpenAI ada-002 | 英文效果好，简单易用 |
| Qwen3-Embedding | 国产，中文优秀 |

**注意：**
- 嵌入模型要和后续检索模型匹配
- 中文场景优先 BGE 系列

---

### 4. 向量数据库

**作用：** 存储文档块的向量，支持快速相似性搜索。

**选型对比：**

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| **pgvector** | PostgreSQL 扩展，简单 | 中小规模（<100万向量），已有 PG 环境 |
| **Qdrant** | Rust 实现，高性能，过滤强 | 需要复杂元数据过滤 |
| **Milvus/Zilliz** | 企业级，支持亿级向量 | 大规模生产 |
| **Weaviate** | 原生混合搜索（向量+关键词） | RAG 场景友好 |
| **Pinecone** | 全托管，零运维 | 快速上线，成本较高 |

**最佳实践：**
- 使用 **混合检索**（向量 + 关键词 BM25）
- 启用 **元数据过滤**（按部门、时间等筛选）
- 加 **重排序（Reranking）** 提升精度

---

### 5. 检索与重排序

**基础检索：**
```
用户问题 → 生成嵌入 → 在向量库中找 Top-K 最相似的块
```

**增强检索：**
- **混合检索**：向量搜索 + 关键词搜索，结果融合
- **重排序**：用交叉编码器对 Top-K 结果重新排序，提升精度
- **元数据过滤**：先按部门/时间筛选，再检索

**效果：** 混合检索 + 重排序可提升 **Precision@5 达 12-19%**

## 四、工具选型对比

### Dify vs FastGPT vs RAGFlow

| 维度 | Dify | FastGPT | RAGFlow |
|------|------|---------|---------|
| **定位** | LLM 应用开发平台 | 轻量知识库 Q&A | 深度文档 RAG 引擎 |
| **RAG 能力** | ★★ 良好 | ★ 基础强 | ★★★ **最强** |
| **文档解析** | 12+ 格式，通用 | 9 种格式 | **DeepDoc，扫描件/表格强** |
| **易用性** | **最友好**，拖拽工作流 | 良好 | 中等，部署稍复杂 |
| **资源需求** | 2核4G（推荐4核8G） | **最轻量** 2核4G | 4核16G+ |
| **GitHub Stars** | 90k+ | ~27k | ~48k |

### 选择建议

| 你的情况 | 推荐 |
|---------|------|
| 文档简单（FAQ、手册）+ 快速上线 | **FastGPT** |
| 需要可视化工作流 + 多种 AI 应用 | **Dify** |
| 文档复杂（合同、报告、扫描件、表格） | **RAGFlow** |
| 想要混合方案 | Dify 做工作流 + RAGFlow 做核心知识库 |

## 五、部署建议

### 硬件配置

| 规模 | 配置建议 |
|------|---------|
| 小团队/测试 | 2核4G，Docker 部署 |
| 中等规模生产 | 4核8G+，SSD |
| 大规模企业 | 8核16G+，GPU（可选），K8s 集群 |

### 部署方式

**最简单 - Docker Compose：**
```bash
# 以 Dify 为例
git clone https://github.com/langgenius/dify.git
cd dify/docker
docker compose up -d
```

**生产环境：**
- 用 Nginx 反向代理
- 配置 HTTPS
- 加监控（日志、延迟、token 消耗）
- 定期备份向量数据库

## 六、最佳实践与避坑指南

### ✅ 推荐做法

1. **文档质量第一**
   - 垃圾进，垃圾出
   - 清洗、去重、结构化后再入库

2. **先用小数据测试**
   - 选 50-100 个典型文档
   - 测试召回率和准确率
   - 调优切块大小和检索策略

3. **添加元数据**
   - 部门、时间、文档类型
   - 支持后续过滤

4. **启用混合检索 + 重排序**
   - 向量 + 关键词
   - Top-K 后重排序

5. **监控和迭代**
   - 记录用户反馈
   - 追踪幻觉案例
   - 持续优化文档和参数

### ❌ 常见坑

| 问题 | 原因 | 解决 |
|------|------|------|
| 回答不相关 | 切块太大/太小 | 调整切块大小，加重叠 |
| 幻觉严重 | 检索不到相关内容 | 优化检索，加混合搜索 |
| 表格/扫描件处理差 | 解析能力不足 | 换 RAGFlow 或专业 OCR |
| 回答太泛 | 文档本身不具体 | 优化知识库内容 |
| 响应慢 | 向量库太大/无缓存 | 加缓存、分层存储 |

## 七、落地方案

### 阶段一：快速验证（1-2 周）
- 用 **FastGPT** 或 **Dify** Docker 部署
- 导入 50-100 个典型文档
- 测试问答效果
- 收集反馈

### 阶段二：生产优化（2-4 周）
- 根据文档复杂度决定是否换 RAGFlow
- 优化切块策略
- 启用混合检索 + 重排序
- 接入内部账号系统（SSO）

### 阶段三：持续迭代
- 监控问答质量
- 定期更新知识库
- 根据反馈优化
